1. Executive summary

Apple Silicon combines CPU, GPU, and Neural Engine (ANE) on one die with unified memory (UMA). Machine learning on this platform is less about “moving data to the GPU” and more about selecting frameworks and compute engines, then validating with Xcode Instruments.

Research & exploration on Mac: MLX (Python/Swift), built on Metal, PyTorch-like APIs.
Shipping inference on device: Core ML + coremltools, compiler routes ops to ANE/GPU/CPU.
Low-level GPU control: Metal (+ Metal Performance Shaders for common primitives).

2. Hardware motivation (Phase 0)

2.1 SoC components (user-level)

  • CPU: Performance cores for latency-sensitive work; efficiency cores for background orchestration.
  • GPU: Massively parallel floating-point and matrix work — training, inference (MLX GPU path), graphics, custom kernels.
  • ANE: Specialized inference accelerator; executes compiler-supported subgraphs, not arbitrary Metal shaders.
  • Unified Memory: One physical RAM pool. CPU, GPU, and ANE reference the same bytes without PCIe-style copies.
  • Die-to-die (D2D): On M-series Ultra, two dies are packaged together. Program as one machine; very large models may be sensitive to cross-die bandwidth — rarely something app code configures directly.

2.2 Why the software stack looks different

  • “Device placement” is often scheduling, not memory migration.
  • Cold-start costs include Metal shader compile and Core ML model compile (cache afterward).
  • Power dominates mobile and always-on workloads — ANE can win even when raw GPU FLOPs are available.

2.3 Phase 0 lab & exit

Lab: Activity Monitor → GPU History; observe idle vs browser vs later MLX/Core ML runs.

Exit: Explain UMA in one paragraph; sketch App → Framework → CPU|GPU|ANE diagram.

4. Metal — GPU interface (Phase 2, ~2 weeks)

4.1 Concepts (in order)

  1. MTLDevice, command queue, command buffer
  2. Buffers/textures with shared storage on Apple silicon
  3. Compute pipelines, threadgroups, occupancy intuition
  4. Completion handlers; avoid unnecessary sync

4.2 Labs

  • A: GPU buffer reduction vs CPU Accelerate; time both.
  • B: Sweep threadgroup sizes; plot rough throughput.
  • C: Xcode Metal debugger + GPU counters on Lab A.

Swift: Use native Metal from Swift for production-shaped code paths.

Exit: Trace Swift → command buffer → compute encoder → GPU; explain why shared buffers avoid copies.

Defer: Ray tracing, mesh shaders, full render pipeline until elective track.

5. GPU ML primitives (Phase 3, ~1 week)

Study MPS and Accelerate as layers below MLX/Core ML. Lab: image decode → MPS normalize → small custom Metal kernel → compare with one MLX op after Phase 5.

6. Core ML & ANE (Phase 4, ~2 weeks)

6.1 Topics

  • .mlpackage lifecycle; first-run compile latency and disk cache
  • Graph partitioning — unsupported ops fall back to GPU or CPU
  • MLTensor and generated Swift interfaces
  • Compute unit preferences: cpuOnly, cpuAndGpu, all

6.2 Tools

  • coremltools conversion from a small public model
  • Xcode: Core ML performance report, Neural Engine instrument, Metal instrument

6.3 Labs

  • Convert same model with different compute preferences; log latency/power.
  • Identify one op that forces GPU fallback from performance report.
  • Minimal Swift app calling MLModel.

Exit: Read a performance report and state likely ANE vs GPU placement.

7. MLX — explore & train (Phase 5, ~2–3 weeks)

7.1 Concepts

  • Lazy evaluation; materialize with eval/async
  • cpu vs gpu as scheduling targets on UMA
  • mlx.nn, optimizers, automatic differentiation
  • mlx-lm for load/generate/fine-tune
  • Custom Metal kernels when mx.fast is insufficient

7.2 Resources

  • MLX documentation
  • WWDC25: Get started with MLX
  • mlx-examples

7.3 Labs

  • Train tiny MLP; watch GPU in Activity Monitor.
  • Run small LLM via mlx-lm; note RAM for weights + KV cache.
  • List which parts of that graph would be poor ANE candidates (conceptual).

Exit: Explain lazy graphs and MLX’s single Metal queue implication for multi-model apps.

9. ANE vs GPU — division of labor (Phase 7)

Critical clarification: ANE does not make GPU streaming multiprocessors (SMs) faster. It executes different, compiled work — often lowering power and freeing the GPU for parallel-friendly ops.
GPUANE
StrengthGeneral parallel compute, training, flexible opsEfficient supported inference subgraphs
API pathMetal, MLX-on-GPUCore ML compiler (primarily)
OverlapCan run concurrent with ANE if pipeline designedGood for steady low-power inference
NotA universal ANE replacementA faster GPU; does not run your Metal kernels

Pipeline insight: Core ML on ANE + MLX on GPU can outperform single-engine MLX for multi-model apps because engines have separate hardware — but watch MLX serialization on one Metal queue vs multi-process tradeoffs.

Exit: Explain when hybrid ANE+GPU pipelines help and when they do not.
